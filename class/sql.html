Oracle SQL Documentation

What is SQL?

Structured Query Language (SQL) is the set of statements with which all programs and users access data in a database. Application programs and tools often allow users access to the database without using SQL directly, but these applications must support SQL when executing the user's request.

How SQL Works

SQL differs from general-purpose programming languages like C and BASIC, and is technically a data sublanguage. The strengths of SQL provide benefits for all types of users, including application programmers, database administrators, managers, and end users. Some features include:

  • Processing sets of data as groups rather than as individual units
  • Providing automatic navigation to the data
  • Using statements that are complex and powerful individually, and that therefore stand alone

SQL lets you work with data at the logical level. Implementation is only necessary for manipulating the data. For example, to retrieve a set of rows from a table, you define a condition used to filter the rows. In this page, all SQL statements use the optimizer, a part of Oracle Database that determines the most efficient means of accessing the specified data.

SQL unifies a variety of tasks into one consistent language, including:

  • Querying data
  • Inserting, updating, and deleting rows in a table
  • Creating, replacing, altering, and dropping objects
  • Controlling access to the database and its objects
  • Guaranteeing database consistency and integrity

All major relational database management systems support SQL, so you can transfer all skills you have gained with SQL from one database to another. In addition, all programs written in SQL are portable. They can often be moved from one database to another with very little modification.

Datatypes

A datatype is either scalar or nonscalar. A scalar type contains a singular value, whereas a nonscalar type contains a set of values. A large object (known as LOB) is a special scalar datatype representing a large scalar value of either binary or character data.

Some datatypes include:

  • Oracle Built-in Datatypes
  • ANSI, DB2, and SQL/DS Datatypes
  • User-Defined Types
  • Oracle-Supplied Types

Some datatypes may require you to use functions, some with multiple arguments, to define the accuracy of data storage. Let's look at the base function of the NUMBER datatype:

NUMBER(p,s)

... where:

  • p is the precision (the total number of significant decimal digits)
  • s is the scale (the number of digits from the decimal point to the least significant digit)

Here are some examples displaying the storage of the NUMBER datatype:

// → User input: 123.45 NUMBER // ← Stored as: 123.45 // → User input: 123.45 NUMBER(3) // ← Stored as: 124 // → User input: 123.45 NUMBER(6, 2) // ← Stored as: 123.45 // → User input: 123.45 NUMBER(6, 1) // ← Stored as: 123.5 // → User input: 123.45 NUMBER(6, -2) // ← Stored as: 100
Literals

The terms literal and constant value are synonymous and refer to a fixed data value. For example, 'JILL', 'BLUE ROCK', and '404' are all character literals; 1010 is a numeric literal. Character literals are enclosed in single quotation marks ('') so that Oracle can distinguish them from schema object names.

Literal types include the following:

  • Text Literals
  • Numeric Literals
  • Datetime Literals
  • Interval Literals

Many SQL statements and functions require you to specify character and numeric literal values. You can also specify literals as part of expressions and conditions.

Below are several examples of valid literals:

// Text Literals 'Goodbye' 'ORACLE.dbs' 'Jessie''s umbrella' '19-APR-98' N'nchar literal' // Numeric Literals 17 6.53 +255 -255 15e-04 // Datetime Literals DATE '1987-12-28' TO_DATE('87-DEC-28 19:30','YY-MON-DD HH24:MI') // Interval Literal INTERVAL '123-2' YEAR(3) TO MONTH
Format Models

A format model is a character literal that describes the format of datetime or numeric data stored in a character string. It's important to note that this does not change the internal representation of the value in the database. Converting a character string into a date or number determines how Oracle Database interprets the string.

For example:

  • The datetime format model for the string '19:30:59' is 'HH24:MI:SS'.
  • The datetime format model for the string '12-Dec-1987' is 'DD-Mon-YYYY'.
  • The number format model for the string '$1,234.56' is '$9,999.99'.
Nulls

If a column in a row has no value, then the column is known to be null, or to contain NULL. Use a NULL when the actual value is not known or when a value would not be meaningful.

All scalar functions (with a few exceptions) return NULL when given a null argument. In SQL, it is possible to return a value (using the NVL function) when a null occurs. For conditions, a NULL value would act similarly to a FALSE value.

Most aggregate functions ignore nulls. For example, consider a query that averages the four values: 2000, NULL, NULL, and 3000. This query ignores the nulls and calculates the average to be (2000+3000)/2 = 2500.

Comments

Comments can increase your application's readability and usability. For instance, including a comment in a statement that describes the purpose of the statement within your application can make it easier to understand. With the exception of hints, statement execution is unaffected by SQL comment lines.

A comment can appear between any keywords, parameters, or punctuation marks in a statement. There are two ways to include comments:

  • Begin the comment with a slash and an asterisk (/*). Comments inside this text can span multiple lines. End the comment with an asterisk and a slash (*/). It is not necessary for the opening and terminating characters to be separated from the text by a space or a line break.
  • Begin the comment with two hyphens (--). Comments inside this text cannot extend to a new line. End the comment with a line break.

Be aware that some of the tools used to enter SQL have additional restrictions.

Here is an example of an application using both kinds of comments:

SELECT last_name, salary + NVL(commission_pct, 0), job_id, e.department_id /* Select all employees whose compensation is greater than that of Priscilla.*/ FROM employees e, departments d /*The DEPARTMENTS table is used to get the department name.*/ WHERE e.department_id = d.department_id AND salary + NVL(commission_pct,0) > /* Subquery: */ (SELECT salary + NVL(commission_pct,0) /* total compensation is salary + commission_pct */ FROM employees WHERE last_name = 'Priscilla'); SELECT last_name, -- select the name salary + NVL(commission_pct, 0), -- total compensation job_id, -- job e.department_id -- and department FROM employees e, -- of all employees departments d WHERE e.department_id = d.department_id AND salary + NVL(commission_pct, 0) > -- whose compensation -- is greater than (SELECT salary + NVL(commission_pct,0)-- the compensation FROM employees WHERE last_name = 'Priscilla'); -- of Priscilla.
Database Objects

Oracle Database recognizes objects that are associated with a particular schema and objects that are not associated with a particular schema. They are known as Schema Objects and Nonschema Objects. Names must be provided for most types of database objects when you create them.

Schema Objects

A schema is a collection of logical structures of data, or schema objects. A schema inherits the name of the database user, where each user owns a single schema. Schema objects can be created and manipulated with SQL and include the following types of objects:

  • Clusters
  • Constraints
  • Database links
  • Database triggers
  • Dimensions
  • External procedure libraries
  • Index-organized tables
  • Indexes
  • Indextypes
  • Java classes, Java resources, Java sources
  • Materialized views
  • Materialized view logs
  • Object tables
  • Object types
  • Object views
  • Operators
  • Packages
  • Sequences
  • Stored functions, stored procedures
  • Synonyms
  • Tables
  • Views
Nonschema Objects

These types of objects are also stored in the database and can be created and manipulated with SQL but are not contained in a schema:

  • Contexts
  • Directories
  • Parameter files and server parameter files
  • Profiles
  • Roles
  • Rollback segments
  • Tablespaces
  • Users
Schema Object Guidelines

Some schema objects are made up of parts that you can or must name, such as the columns in a table or view, index and table partitions and subpartitions, integrity constraints on a table, and objects that are stored within a package, including procedures and stored functions.

Schema Object Naming Rules

Every database object has a name. In a SQL statement, you represent the name of an object with either a quoted or a nonquoted identifier.

  • A quoted identifier begins and ends with double quotation marks ("). If you name a schema object using a quoted identifier, then you must use the double quotation marks whenever you refer to that object.
  • A nonquoted identifier is not surrounded by any punctuation.

You can use either quoted or nonquoted identifiers to name any database object. However, database names, global database names, and database link names are always case insensitive and are stored as uppercase. If you specify such names as quoted identifiers, then the quotation marks are silently ignored.

The following list of rules applies to both quoted and nonquoted identifiers unless otherwise indicated:

  1. Names must be from 1 to 30 bytes long with these exceptions:
    • Names of databases are limited to 8 bytes.
    • Names of database links can be as long as 128 bytes.

    If an identifier includes multiple parts separated by periods, then each attribute can be up to 30 bytes long. Each period separator, as well as any surrounding double quotation marks, counts as one byte.

  2. Nonquoted identifiers cannot be Oracle Database reserved words. Quoted identifiers can be reserved words, although this is not recommended.

    3. Names may be further restricted by other product-specific reserved words depending on the used Oracle product.

  3. Do not reuse names that the Oracle SQL language uses as special cases. Although they are not reserved, Oracle uses them internally in specific ways. These include datatypes, schema names, function names, the dummy system table DUAL, and keywords (the uppercase words in SQL statements, such as DIMENSION, SEGMENT, ALLOCATE, DISABLE, and so forth). These words are not reserved. If you use these words as names for objects and object parts, then your SQL statements may be more difficult to read and may lead to unpredictable results.
  4. It is strongly advised to use ASCII characters in database names, global database names, and database link names.
  5. Nonquoted identifiers must begin with an alphabetic character from your database character set, while quoted identifiers can begin with any character.
  6. Nonquoted identifiers can contain only alphanumeric characters from your database character set and the underscore (_), dollar sign ($), and pound sign (#). Database links can also contain periods (.) and "at" signs (@). It is strongly advised to not use $ and # in nonquoted identifiers.
  7. Within a namespace, no two objects can have the same name. Only certain schema and nonschema objects may share the same namespace.
  8. Nonquoted identifiers are not case sensitive, as Oracle interprets them as uppercase. Only quoted identifiers are case sensitive.
  9. Columns in the same table or view cannot have the same name; however, columns in different tables or views can have the same name.
  10. Procedures or functions contained in the same package can have the same name, but only if their arguments are not of the same number and datatypes.
Schema Object Naming Examples

The following examples are valid schema object names:

first_name rabbit hr.hire_time "EVEN THIS & THAT!" a_very_long_but_valid_name

All of the examples above adhere to the rules listed in "Schema Object Naming Rules".

The example below is not valid because it exceeds 30 characters:

a_very_very_long_but_not_valid_name
Schema Object Naming Guidelines

To assist in following proper naming conventions, here are helpful guidelines for naming objects and their parts:

  • Use full, descriptive, pronounceable names (or well-known abbreviations).
  • Use consistent naming rules.
  • Use the same name to describe the same entity or attribute across tables.

It is important to balance the objective of keeping names short and easy to use with the objective of making names as descriptive as possible. When in doubt, choose the more descriptive name, because the objects in the database may be used by many people over a period of time.
